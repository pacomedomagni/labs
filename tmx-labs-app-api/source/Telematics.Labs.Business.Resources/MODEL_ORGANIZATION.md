# Model Organization Strategy

This document outlines the model organization strategy for the Telematics Labs API.

## Architecture Pattern: Full Separation

This project uses **full separation** between API contracts and domain models:
- Controllers receive API Request models
- Controllers map requests to Domain models
- Orchestrators work with Domain models only
- Controllers map Domain results to API Response models

This provides clean boundaries but requires mapping at the controller level.

## Model Types and Locations

### 1. API Request Models
**Location:** `Telematics.Labs.Api/RequestModels/{Feature}/`  
**Naming Convention:** `*Request`  
**Purpose:** Define the contract for incoming API requests

```
Telematics.Labs.Api/
??? RequestModels/
    ??? Fulfillment/
        ??? AssignDevicesRequest.cs
        ??? GetOrdersByStatusRequest.cs
```

**Guidelines:**
- Keep in the API project for controller-specific validation
- Use `[Required]` and other data annotations
- Reference domain models when needed for complex nested objects

### 2. API Response Models
**Location:** `Telematics.Labs.Api/ResponseModels/{Feature}/`  
**Naming Convention:** `*Response`  
**Purpose:** Define the contract for API responses

```
Telematics.Labs.Api/
??? ResponseModels/
    ??? Fulfillment/
        ??? AssignDevicesResponse.cs
```

**Guidelines:**
- Keep in the API project for full separation from domain
- Controller maps domain model ? API response
- Can include API-specific formatting attributes

### 3. Business/Domain Models
**Location:** `Telematics.Labs.Business.Resources/Domain/{Feature}/`  
**Naming Convention:** No suffix - use descriptive domain names  
**Purpose:** Internal business logic representation

```
Telematics.Labs.Business.Resources/
??? Domain/
?   ??? Fulfillment/
?       ??? FulfillmentOrderDetails.cs
?       ??? FulfillmentVehicle.cs
?       ??? FulfillmentDeviceType.cs
?       ??? DeviceOrderInfo.cs
??? Resources/              # Legacy API response models (for backward compat)
?   ??? FulFillment/
??? Enums/
```

**Guidelines:**
- Contains business logic and validation
- Should not contain API-specific concerns (e.g., JSON attributes)
- Orchestrators return these types

### 4. DAL (Data Access Layer) Models
**Location:** `Telematics.Labs.Services/Database/Models/{Feature}/`  
**Naming Conventions:**
- `*Entity` - Direct database table/view mapping
- `*QueryResult` or `*DataModel` - Complex query projections
- `*Params` - Stored procedure input parameters

```
Telematics.Labs.Services/
??? Database/
    ??? Models/
        ??? DeviceOrder/
            ??? FulfillmentOrderEntity.cs      # Query result from SP
            ??? DeviceOrderCreationResult.cs   # SP output
            ??? CreateReplacementDeviceOrderModel.cs  # SP params
```

**Guidelines:**
- Map directly to database columns/stored procedure results
- No business logic
- Use Dapper-compatible types

## Mapper Locations

### Business Layer Mappers (`Telematics.Labs.Business/Mappers/`)
Handle transformations that don't involve API types:
- DAL Entity ? Domain Model
- API Request Model ? Domain Model (when request types are in Business.Resources)

```csharp
// FulfillmentMappers.cs
public static FulfillmentOrderDetails ToFulfillmentOrderDetails(this OrderDetailsModel model);
public static FulfillmentVehicle ToFulfillmentVehicle(this MyScoreVehicle vehicle);
public static DeviceOrderInfo ToDeviceOrderInfo(this FulfillmentOrderEntity entity);
```

### API Layer Mappers (`Telematics.Labs.Api/Mappers/`)
Handle transformations that produce API response types:
- Domain Model ? API Response

```csharp
// FulfillmentResponseMappers.cs
public static AssignDevicesResponse ToAssignDevicesResponse(this FulfillmentOrderDetails domain);
public static VehicleAssignmentResponse ToVehicleAssignmentResponse(this FulfillmentVehicle domain);
```

## Controller Pattern

```csharp
[HttpPost("AssignDevices")]
public async Task<ActionResult<AssignDevicesResponse>> AssignDevicesToOrder(
    [FromBody] AssignDevicesRequest request)
{
    // 1. Map API request ? Domain models
    var orderDetails = request.OrderDetails.ToFulfillmentOrderDetails();
    var vehicle = request.MyScoreVehicle.ToFulfillmentVehicle();

    // 2. Call orchestrator with domain models
    var domainResult = await _orchestrator.AssignDevices(orderDetails, vehicle);

    // 3. Map domain result ? API response
    var response = domainResult.ToAssignDevicesResponse();
    return Ok(response);
}
```

## Migration Strategy

When updating existing endpoints to full separation:

1. **Create API Response model** in `Api/ResponseModels/{Feature}/`
2. **Create/verify Domain model** in `Business.Resources/Domain/{Feature}/`
3. **Add API mapper** in `Api/Mappers/` for Domain ? Response
4. **Add Business mapper** in `Business/Mappers/` for Request ? Domain (if needed)
5. **Add new orchestrator method** returning Domain model
6. **Mark old orchestrator method** with `[Obsolete]`
7. **Update controller** to use new pattern
8. **Remove old method** in future release
